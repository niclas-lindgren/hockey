# Plan: Full-season tournament schedule generator across nine RVV clubs
**Goal:** Given a roster config and season window, the tool scrapes all club calendars (existing + new), checks conflicts, and outputs a proposed set of 10-15 tournament dates with assigned matchups, optimizing for matchup diversity and even time spread, viewable in console and exportable to Excel.
**Created:** 2026-06-08
**Intent:** Replace ad-hoc per-tournament date hunting with a single end-to-end planner that can produce a whole season's worth of tournament proposals across all nine Region Viken Vest (RVV) clubs at once.

## Notes
- The existing single-tournament flow (`TournamentScheduler.find_available_dates`, the CLI's `--reschedule` mode, etc.) stays as its own ad-hoc tool — it's used for one-off rescheduling/replanning during the season and is NOT being replaced or repurposed as the foundation of region-wide season planning. Reuse only the shared lower layers — `fetch_events`/`CalendarCache`/conflict-checkers/club_registry — not the single-tournament orchestration logic itself. `SeasonPlanner` calls `find_available_dates` purely to obtain the set of conflict-free candidate dates for the season window; it does not extend or modify that class.
- All console output must go through `TournamentOutput` (Rich) per project convention — avoid raw `print`.
- Club URLs/calendar IDs are currently hardcoded inline in tournament_scheduler.py; the new club registry should become the single source of truth so the seven new clubs (Holmen, Jutul, Jar, Ringerike, Frisk Asker, Tønsberg, Sandefjord Penguins) plug in the same way Kongsberg/Skien do today — together with Kongsberg/Skien, all 9 RVV clubs (https://www.rvvhockey.no/).
- Use best judgment between a simple greedy/round-robin diversity heuristic and a more sophisticated optimization — a greedy least-recently-played assignment is sufficient to satisfy the goal without a large rewrite.
- Tournaments rotate across arenas (one per club) — every arena should host at least one tournament across the season. Teams belong to age groups (boys: U7-U12; girls: JU10/JU11/...). Age groups whose player pools overlap (e.g. JU11 and U10) should preferably avoid sharing a tournament weekend — treat this as a soft constraint the planner optimizes for and reports on, not a hard failure.
- Calendar URLs for Jutul, Jar, and Ringerike were provided directly by the user (see club registry task and PROJECT.md "Season-scheduling extension" section); Holmen, Frisk Asker, Tønsberg, and Sandefjord Penguins URLs are still missing — Sandefjord Penguins already appears as an opponent in the existing Excel schedule (`existing_schedule/U10_ETTER_JUL_Klar_-_Kongsberg_Sandefjord.xlsx`) but has no scraper.
- Spond export (Excel-import format for season planning) is explicitly out of scope for this plan — the user asked for it last, after the season-overview/per-tournament/round-robin output is solid. It is tracked separately in BACKLOG.md as a follow-up item once this plan ships.
- No RESEARCH.md or HISTORY.md existed at plan time; findings above come from direct codebase inspection (scheduler.py, interfaces.py, data_sources/, conflict_checkers/, excel/tournament_reader.py, utils/rich_output.py, utils/calendar_cache.py, tournament_scheduler.py, tournament_scheduler_interactive.py).

## Acceptance Criteria
- [ ] Running the season-generation flow with a roster config (e.g. Jar 1, Jar 2, plus teams from the other clubs/age groups) and an Oct-Apr window prints a Rich season-overview table listing 10-15 proposed tournaments — each with date, age group/gender, arena, and participating teams.
- [ ] For any tournament in the plan, `print_tournament_schedule` shows a full round-robin game schedule where every participating team plays every other participating team exactly once, with concurrent games grouped into parallel slots according to that age group's `parallelGames` setting from the config (e.g. `{U7: {parallelGames: 4}, U10: {parallelGames: 3}}`).
- [ ] The proposed plan varies *which teams get grouped into the same tournament together* across the season — since round-robin guarantees full pairing within a tournament, diversity is about invitation groupings, not pairwise tracking: the diversity summary shows that no team is repeatedly invited alongside the same handful of clubs every time, by a wide margin.
- [ ] `SeasonPlan` metadata reports a per-arena tournament count that is ≥ 1 for every arena/club in the roster, and the diversity summary prints any unavoidable same-weekend collisions between overlapping age groups (e.g. JU11 vs U10) so the organizer can review them.
- [ ] Passing `--export-excel <path>` (or the interactive equivalent) writes an `.xlsx` file that, when read back with openpyxl, contains a season-overview sheet matching the console-reported tournaments and per-tournament sheets matching each tournament's round-robin game list.
- [ ] The club registry returns a working `CalendarDataSource` for each of the nine RVV clubs (Ringerike, Tønsberg, Frisk Asker, Sandefjord Penguins, Jar, Holmen, Skien, Jutul, Kongsberg), and `pytest tests/test_club_registry.py` passes.
- [ ] `pytest tests/test_season_planner.py tests/test_round_robin.py tests/test_plan_exporter.py` exits with code 0, confirming the planner produces 10-15 evenly-spread tournaments, the round-robin generator produces correct complete game schedules respecting per-age-group parallel-game limits, and the exporter round-trips data through Excel correctly.
